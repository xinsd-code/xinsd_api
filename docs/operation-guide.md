## 1. 环境准备

### 1.1 首次启动

```bash
# 确保 Node.js >= 20
node -v

# 安装依赖
npm install

# 复制环境配置
cp .env.example .env.local

# 启动开发服务器
npm run dev
# 访问 http://localhost:3000
```

### 1.2 配置 AI 模型（必须）

> 在使用 NL2DATA、DB Harness、AI Chat 编排等功能前，需要先配置至少一个 AI 模型。

**操作步骤**：

1. 访问 **模型管理** 页面（`/model-management`）
2. 点击"新增模型来源"
3. 填写以下信息：
   - **名称**：模型来源标识（如"DeepSeek"）
   - **Base URL**：模型 API 地址（如 `https://api.deepseek.com`）
   - **鉴权方式**：选择 Bearer Token 或自定义 Header
   - **API Key**：填写你的 API 密钥
4. 添加 **Model ID**（如 `deepseek-chat`），系统会实时校验可用性
5. 保存后，该模型即可在所有 AI 功能中使用

### 1.3 配置数据库实例（NL2DATA / DB Harness 必须）

**操作步骤**：

1. 访问 **数据库实例** 页面（`/database-instances`）
2. 点击"新增实例"
3. 填写连接信息：
   - **类型**：MySQL / PostgreSQL / MongoDB / Redis
   - **主机**：数据库地址
   - **端口**：数据库端口
   - **用户名 / 密码**
   - **数据库名**
4. 点击"测试连接"确认连通性
5. 保存后，进入实例详情可浏览表结构

### 1.4 配置语义信息（推荐）

> 语义配置是提升 NL2DATA 和 DB Harness 准确率的关键。配置越完善，AI 的字段映射和 SQL 生成越精准。

**操作步骤**：

1. 进入 **数据库实例** 详情页
2. 选择需要配置的表
3. 逐字段编辑语义信息：
   - 设置 **语义名称**（如 `order_amount` → "订单金额"）
   - 添加 **别名**（如 "订单额"、"交易金额"）
   - 选择 **语义角色**（metric / dimension / time / identifier / attribute）
   - 开启 **NER 开关**（纳入自然语言识别候选）
4. 保存后系统自动生成规范化映射

**最佳实践**：

- 优先配置核心业务表（订单、用户、商品等）
- 为高频查询字段开启 NER
- 添加尽可能多的别名，涵盖口语化表达
- 为指标字段填写计算口径说明

## 2. API 高级编排操作指南

### 2.1 创建转发配置

1. 访问 **API 转发** 页面（`/api-forward`）
2. 点击"新增转发"
3. 配置基本信息：
   - **请求方法**：GET / POST / PUT / DELETE
   - **虚拟路径**：如 `/products/list`
   - **目标类型**：Mock 接口 / API Client / DB API
   - **目标接口**：选择已有的目标接口
4. 配置参数绑定（可选）：
   - **入参绑定**：将调用方传入的参数映射到目标接口
   - **固定值**：设置固定静态值
   - **Body 注入**：将参数写入请求体字段
5. 保存

### 2.2 添加编排节点

1. 在转发配置详情中，找到编排工作区
2. 点击"添加节点"
3. 选择节点类型并配置：

**filter 节点（字段筛选）**：
```
模式：include（保留）或 exclude（排除）
字段列表：输入路径，支持嵌套（如 data[].id, data[].name）
```

**map 节点（字段重命名）**：
```
映射规则：
  原路径 → 新路径
  例：data[].price → data[].amount
```

**compute 节点（计算字段）**：
```
目标字段：如 data[].discountPrice
表达式：{{amount}} * 0.9
或直接引用：sourceField = $param.taxRate
```

**sort 节点（排序限制）**：
```
数组路径：data
排序字段：discountPrice
排序方向：降序（desc）
限制数量：10
```

4. 按需调整节点顺序（order 值越小越先执行）
5. 保存编排配置

### 2.3 使用 AI Chat 生成编排

1. 点击编排工作区右上角的 **AI Chat** 按钮
2. 用自然语言描述需求，例如：
   - "只保留 data 数组中的 id、name、price 字段"
   - "把 price 字段重命名为 amount，然后计算一个 discountPrice = amount * 0.9"
   - "按 discountPrice 降序排列，只取前 10 条"
3. AI 返回的 JSON 配置会自动应用到编排画布
4. 如有体检报错，可使用"修复体检报错"模式让 AI 最小改动修复

### 2.4 调试编排节点

1. 先执行一次目标接口获取响应数据
2. 在编排工作区选择需要调试的节点
3. 点击"调试"按钮
4. 系统会执行到该节点为止，显示中间输出结果
5. 验证数据变换是否符合预期

### 2.5 访问编排后的接口

```bash
# 直接调用虚拟路径即可：
curl "http://localhost:3000/forward/products/list?page=1&size=10"

# 编排配置存在时，返回的是经过编排处理后的数据
```

## 3. NL2DATA 操作指南

### 3.1 基本取数流程

1. 访问 **NL2DATA** 工作台（`/nl2data`）
2. 在右上角选择：
   - **数据源**：目标数据库实例
   - **模型**：用于推理的 AI 模型
3. 在输入框输入自然语言问题，例如：
   - "查看近 7 天各渠道的订单金额"
   - "统计上月各城市的用户注册数量，按注册数降序排列"
   - "查找最近 30 天订单金额 TOP10 的客户"
4. 点击"执行"
5. 等待 AI 两阶段处理：
   - 第一阶段：NER 识别业务名词
   - 第二阶段：生成 SQL
6. 查看结果：
   - SQL 编辑区显示生成的 SQL
   - 数据表格显示查询结果
   - Trace 面板显示各阶段执行状态

### 3.2 手动调整 SQL

- SQL 编辑区可以手动修改 AI 生成的 SQL
- 修改后点击"执行"重新运行
- 重复执行相同 SQL 不会产生新的历史条目

### 3.3 使用会话历史

- 左侧会话历史面板显示所有历史记录
- 点击某条记录可查看：
  - 用户原始问句
  - 生成的 SQL
  - 完整的 Prompt（用于调试）
- 点击"同步 SQL"可将该 SQL 回填到编辑区

### 3.4 最佳实践

| 建议 | 说明 |
|------|------|
| 明确时间范围 | "近 7 天"、"上月"、"2024年Q1"比"最近的"更精确 |
| 指明指标和维度 | "各渠道的订单金额"比"看看订单数据"更容易命中 |
| 配置语义 | 提前配置语义名称和别名，大幅提升 NER 匹配准确率 |
| 逐步追问 | 先问简单问题，再基于结果追问细化 |
| 选择合适模型 | 推理能力强的模型（如 deepseek-chat）效果更好 |

## 4. DB Harness 操作指南

### 4.1 创建 Workspace

1. 访问 **DB Harness** 页面（`/db-harness`）
2. 在左侧面板点击"新建 Workspace"
3. 填写 Workspace 名称
4. **绑定数据源**：选择目标数据库实例
5. **设置工作规则**（可选但推荐）：
   ```
   示例规则：
   1. 所有查询默认限制 100 行
   2. 时间字段使用 created_at
   3. 用户相关查询排除测试用户（user_type != 'test'）
   4. 金额字段单位为"分"，展示时需除以 100
   ```
6. 保存 Workspace

### 4.2 开始对话

1. 选择或新建 Session
2. 在右上角选择 AI 模型
3. 输入问题开始对话，例如：
   - "近 7 天各渠道的 GMV 对比"
   - "本月用户注册数趋势"
   - "分析订单取消率较高的原因"
4. 等待五阶段 Agent 依次执行
5. 查看结果：
   - **回复区**：AI 对数据结果的自然语言解读
   - **数据表格**：原始查询结果
   - **Trace 面板**：五阶段执行链路
   - **追问建议**：3 个智能追问方向

### 4.3 查看 Trace 链路

每轮对话可以展开 Trace 面板查看五个阶段的详情：

| Stage | 查看内容 |
|-------|---------|
| Intent Agent | 识别的意图类型（query/comparison/analysis）、规划提示 |
| Schema Agent | NER 匹配结果、已映射字段列表、未匹配术语 |
| Query Agent | 查询计划（维度/指标/筛选/排序）、生成的 SQL |
| Guardrail Agent | 安全校验状态、执行结果预览（行数/列名） |
| Analysis Agent | 自然语言回复、结果摘要、追问建议 |

点击 Handoff 可查看每个 Agent 传给下一阶段的完整上下文。

### 4.4 提交知识反馈

- **点赞好结果**：点击 👍 标记当前回答为"可复用"知识
- **纠错差结果**：点击 👎 并填写纠错说明，标记为"需优先纠偏"
- 反馈写入知识库后，后续相同类型的问题会优先参考历史知识

### 4.5 运行 GEPA 评估

> GEPA（Generative Evaluation and Policy Adaptation）可以离线评估并优化 Prompt 策略。

1. 在 Workspace 设置中找到 GEPA 入口
2. 点击"创建评估 Run"
3. 确认评估参数：
   - **样本数量**：1-20 条（默认 12）
   - **Prompt 候选数**：默认 2
   - **策略候选数**：默认 3
4. 系统自动：
   - 提取历史会话样本（不足时合成样本）
   - 对比各候选策略的评分
   - 生成评分卡和排名报告
5. 审核报告中的候选排名
6. 点击"应用"将最优配置写入 Workspace

### 4.6 最佳实践

| 建议 | 说明 |
|------|------|
| 配置工作规则 | 明确时间字段、默认限制、排除规则等业务约束 |
| 完善语义配置 | 核心业务字段配好语义名称、别名、角色 |
| 积极反馈 | 多点赞好结果、纠错差结果，帮助知识库积累 |
| 分 Workspace 管理 | 不同数据分析场景/数据源使用不同 Workspace |
| 追问而非重问 | 利用上下文连续追问，比重新开始更高效 |
| 关注 Trace | 遇到错误先查看 Trace，定位是哪个 Agent 出了问题 |
| 定期运行 GEPA | 随着知识积累，定期评估优化 Prompt 策略 |

## 5. 通用配置操作

### 5.1 Redis 缓存配置（可选）

适用于 API 转发和 DB API，在 `.env.local` 中配置：

```env
REDIS_URL=redis://localhost:6379
```

在接口配置中开启"Redis 缓存"并设置：
- **缓存 Key 模板**：支持 `{{param.key}}` 和 JSONPath 动态拼接
- **过期时间**：默认不设置（永不过期）

### 5.2 生产部署

```bash
# 构建
npm run build

# 启动
npm start

# 或使用 PM2 托管
pm2 start ecosystem.config.js --only xinsd-api
```

环境变量配置（`.env.production`）：

```env
NODE_ENV=production
PORT=3000
HOSTNAME=0.0.0.0
DATA_DIR=/var/lib/api-forge
```

### 5.3 健康检查

```bash
curl http://localhost:3000/api/health
```

## 6. 常见问题排查

### Q1: NL2DATA / DB Harness 提示"请先配置模型"

**原因**：未配置 AI 模型来源  
**解决**：前往 `/model-management` 配置至少一个可用的模型来源

### Q2: 数据库连接失败

**原因**：数据库实例配置错误或服务未启动  
**解决**：
1. 确认数据库服务已启动
2. 检查连接地址、端口、用户名、密码
3. 在数据库实例管理页点击"测试连接"验证

### Q3: NER 识别不到业务名词

**原因**：语义配置不完善  
**解决**：
1. 为目标字段配置语义名称和别名
2. 开启 NER 开关
3. 添加多种口语化表达作为别名

### Q4: AI 生成的 SQL 字段名错误

**原因**：模型"幻觉"，引用了不存在的字段  
**解决**：
1. Guardrail Agent 会自动校验字段合法性并阻断
2. 完善语义配置，让 AI 有更准确的 Schema 上下文
3. 在 DB Harness 中提交纠错反馈，记录正确映射

### Q5: 编排节点执行后数据不符预期

**原因**：节点配置路径错误或顺序不当  
**解决**：
1. 使用"单节点调试"功能，逐节点检查中间输出
2. 注意 map 节点重命名后，后续节点需使用新路径
3. 数组操作使用 `data[].field` 语法

### Q6: 模型请求超时

**原因**：模型响应慢或网络不稳定  
**解决**：
1. 系统默认超时 25 秒
2. 超时后自动降级到 Fallback 规则引擎
3. 可切换到响应更快的模型
4. 检查网络与模型服务可用性

### Q7: DB Harness 查询返回空结果

**原因**：筛选条件过严或时间范围过窄  
**解决**：
1. 开启环境变量 `DB_HARNESS_EMPTY_RESULT_RETRY=1` 启用自动重试
2. 手动放宽时间范围或减少筛选条件
3. 参考 Analysis Agent 提供的智能提示
