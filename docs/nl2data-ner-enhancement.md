## NL2DATA 名词识别增强方案

### 背景

当前 NL2DATA 直接将用户问句送入 SQL 生成 prompt。对于业务名词、字段别名、指标叫法较多的数据源，这种单阶段方式容易出现：

- 名词识别不稳定
- 字段映射偏差
- SQL 结果与问句主语义不一致

为提升 SQL 准确率，NL2DATA 改造为“两阶段”链路：

1. 第一阶段进行 NER 名词识别
2. 第二阶段基于 NER 结果生成 SQL

### 指标配置扩展

在数据库实例的字段级指标配置中新增：

- `enableForNer?: boolean`
- `aliases?: string[]`

含义：

- `enableForNer`
  - 控制该字段是否参与 NL2DATA 的第一轮名词识别
  - 默认不参与
- `aliases`
  - 预留的业务别名能力
  - 第一版可先支持数据结构，不强制开放复杂 UI

第一版 UI 最小实现：

- 在“指标配置”页每个字段增加“参与名词识别”勾选项
- 勾选后，该字段的指标名称、说明、字段名、表名可参与 NER 语料构建

### 两阶段链路

#### 阶段一：NER 名词识别

职责：

- 从用户问句中提取核心业务名词
- 将问句中的名词与指标配置、字段、表做映射
- 输出结构化 JSON，不生成 SQL

建议输出：

```json
{
  "normalizedTerms": ["模型", "用户问句", "近7天"],
  "matchedMetrics": [
    {
      "term": "模型",
      "table": "chat_model",
      "column": "model",
      "metricName": "模型名称",
      "confidence": "high"
    }
  ],
  "unmatchedTerms": ["通义系列"],
  "timeHints": ["近7天"],
  "intent": "query"
}
```

#### 阶段二：SQL 生成

职责：

- 基于原始问句、当前 SQL、结果摘要、schema 摘要和第一轮 NER 结果生成最终 SQL
- 优先参考第一轮命中的实体
- 在低置信度场景下保守生成，不强行绑定错误字段

### 第一轮 NER 的控长策略

第一轮 prompt 不能直接注入全量勾选名词库，否则上下文会迅速膨胀，导致识别准确率下降。

#### 核心原则

1. 全量词库只参与本地召回，不直接进入 prompt
2. 第一轮 prompt 只吃高相关候选实体
3. 候选实体必须做去重、归并、压缩和限额裁剪
4. 第一轮 prompt 以短词卡片为主，不承载长描述和完整 schema

#### 语料来源

仅从 `enableForNer = true` 的字段中收集：

- 指标名称 `metricName`
- 指标描述 `description`
- 预留别名 `aliases`
- 字段名
- 表名

#### 本地召回

服务端先根据用户问句做轻量打分召回，建议规则：

- 命中 `metricName`：`+5`
- 命中 `aliases`：`+4`
- 命中字段名：`+3`
- 命中表名：`+2`
- 命中 `description`：`+1`
- 位于高相关表：`+2`

#### 去重归并

两层处理：

- 词面去重：同名词只保留一次
- 语义归并：同一 `table + column` 的多种叫法合并为一个候选实体

#### 压缩规则

第一轮每个候选只保留：

- `table`
- `column`
- `metricName`
- `aliases` 前 2 到 3 个
- `description` 的短摘要

长描述必须截断。

#### 硬性预算

推荐预算：

- 候选字段最多 `12~20`
- 候选表最多 `3~5`
- 第一轮动态上下文控制在 `1500~2500 chars`

超预算时按以下顺序裁剪：

1. 删除低分候选
2. 截断长描述
3. 截断别名
4. 仅保留强匹配项

### 第一轮 Prompt 结构

建议以结构化 JSON 上下文注入：

```json
{
  "question": "最近7天调用次数最多的模型有哪些？",
  "candidateCount": 16,
  "totalAvailable": 143,
  "truncated": true,
  "candidates": [
    {
      "table": "chat_model",
      "column": "model",
      "metricName": "模型名称",
      "aliases": ["模型", "model"],
      "description": "模型名称字段"
    }
  ]
}
```

模型规则：

- 仅做名词识别
- 不生成 SQL
- 优先从候选集中映射
- 无法确认时写入 `unmatchedTerms`

### 第二轮 Prompt 结构

第二轮 prompt 输入建议包括：

- 原始问句
- 当前 SQL 草稿
- 当前结果摘要
- 第一轮 NER 结果
- 精简后的 schema 摘要
- 数据源类型和实例信息

第二轮规则：

- 优先参考 `matchedMetrics`
- 如果 NER 结果为空，允许继续生成 SQL，但标记为低置信度
- 避免忽略时间语义和核心业务实体

### 失败兜底

建议三类兜底：

1. 第一轮无有效匹配
   - 第二轮继续保守生成 SQL
2. 第一轮候选过多
   - 强制 Top N 裁剪，不进入全量模式
3. 第二轮 SQL 偏离第一轮实体
   - 后续可增加轻量校验和重试

### 验收建议

建议覆盖以下场景：

- 指标名直命中
- 指标描述命中
- 多候选冲突
- 大词库场景下 Top N 裁剪
- 第一轮无命中但第二轮仍可生成 SQL
- 指标配置页勾选开关与 NER 实际行为一致
