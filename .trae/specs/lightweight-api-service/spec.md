# 轻量级API服务 - 产品需求文档

## 1. 产品概述
### 1.1 产品定位
一款基于Web的API接口管理、转发与模拟平台。允许开发者灵活接入第三方API，自定义本地API路由，并通过配置实现请求的透明转发、参数转换或直接返回Mock数据。

### 1.2 核心价值
*   **对前端：** 解决跨域问题（CORS）；在后端未开发完时，提供高度真实的Mock数据。
*   **对后端：** 统一管理第三方API（如支付、短信、AI接口），隐藏敏感的API Key，避免在客户端暴露。
*   **对团队：** 提供类似Postman的在线调试体验，且无需下载客户端。

## 2. 核心功能架构设计

整个系统可以划分为五个核心模块：
1.  **项目与环境管理：** 组织API资源，区分开发/测试/生产环境。
2.  **第三方API资源库（目标源）：** 录入和管理所有需要接入的外部接口。
3.  **本地API网关配置（暴露端）：** 定义对外提供的接口路径及出入参。
4.  **路由与处理引擎（核心）：** 处理“本地API -> 第三方API”的转发逻辑，或触发Mock引擎。
5.  **在线调试与日志：** 提供请求发送面板和详细的调用日志。

## 3. 详细功能需求

### 3.1 模块一：项目与环境管理
为了支持个人使用和多场景需求，需要基础的组织机制。
*   **项目空间 (Workspace)：** 用户可以创建多个项目，项目内包含独立的API集合。
*   **API分组功能：** 支持在项目内对API进行分组管理，方便组织和查找。
*   **环境变管理 (Environments)：**
    *   允许定义全局变量和环境变量（如 `{{BASE_URL}}`, `{{API_KEY}}`）。
    *   支持多环境切换（如 Dev, Test, Prod）。

### 3.2 模块二：第三方API接入 (上游服务/Upstream)
类似Postman的接口定义能力，用于提前录入第三方接口。
*   **基本信息：** 请求方法 (GET, POST等)、URL、接口名称、描述。
*   **请求鉴权 (Auth)：** 支持 Bearer Token, API Key, Basic Auth 等，**可引用环境变量**。
*   **请求参数配置：** Headers, Query Params, Path Variables, Body (JSON/Form-data/XML)。

### 3.3 模块三：本地API配置 (下游服务/Downstream) 【核心需求】
用户在此自定义他们想要在本地（或你的平台上）暴露出去的接口。
*   **自定义基础信息：**
    *   自定义路径 (例如: `/my-api/v1/get-weather`)。
    *   请求方法设定。
    *   入参定义（可选：对请求参数进行校验，如必填项、数据类型等）。
*   **处理模式选择 (Mode)：** 每个本地API必须选择一种处理模式：
    *   **模式A：直接Mock (Mock Mode)**
    *   **模式B：请求转发 (Proxy Mode)**

### 3.4 模块四：处理引擎配置 【核心亮点优化】
根据上述选择的模式，进行详细配置。

#### 模式A：直接Mock (Mock Mode)
*   **静态Mock：** 用户直接填入一段JSON，请求该本地接口时直接返回该JSON。
*   **动态Mock (优化建议)：** 集成 `Faker.js` 或类似库，支持语法生成随机数据（如 `@cname` 生成随机中文名，`@email` 生成随机邮箱）。
*   **条件Mock (高级优化)：** 根据入参返回不同数据。例如：入参 `status=1` 返回成功JSON，`status=2` 返回错误JSON。

#### 模式B：请求转发 (Proxy Mode)
*   **目标绑定：** 选择该本地API要转发给哪个已录入的【第三方API】。
*   **请求参数映射 (Request Mapping) - *非常重要*：**
    *   **透传模式：** 将客户端发来的所有参数、Header原封不动转发给第三方。
    *   **转换模式：** 允许将本地入参的字段映射为第三方的字段。例如：本地请求传 `userId`，转发给第三方时自动变成 `uid`。
    *   **参数注入：** 允许在转发时，网关自动加上某些参数或Header（如在这里注入第三方的 `Secret-Key`，从而不在前端暴露）。
*   **响应处理 (Response Transformation)：**
    *   允许将第三方返回的数据进行包装或提取后再返回给调用方。（例如：第三方返回很臃肿的XML，网关将其转换为精简的JSON返回）。

### 3.5 模块五：在线调试面板 (类似Postman界面)
*   **发送请求：** 提供一个UI界面，可选择本地API，填写参数并点击“Send”发送请求。
*   **响应展示：** 友好的高亮展示 Response Body, Headers, HTTP Status Code, 耗时 (Latency)。
*   **一键生成代码：** 支持将当前请求转换为 cURL, Fetch, Axios, Python 等代码片段。

### 3.6 模块六：日志与监控
*   **请求追溯 (Trace)：** 记录每一次对“本地API”的请求日志。
*   **转发详情：** 在日志中能清晰看到：客户端发来的原始请求 -> 网关转换后的请求 -> 发给第三方的请求 -> 第三方的响应 -> 网关转换后返回给客户端的响应。（这对于排查Bug极其重要）。

## 4. 非功能性需求 (技术与体验保障)

*   **安全性 (Security)：**
    *   **防 SSRF 攻击：** 因为你的平台有代发请求的功能，必须严格限制用户不能将目标URL设置为你的内网IP（如 `127.0.0.1`, `192.168.x.x` 等），防止被当作内网扫描跳板。
    *   **凭证加密：** 用户填写的第三方 API Secret 等敏感信息，在数据库中必须加密存储。
*   **性能 (Performance)：**
    *   转发请求的中间损耗应控制在 `20ms` 以内，支持高并发处理。
*   **易用性 (Usability)：**
    *   UI设计需左右分栏（左侧目录树，右侧配置/调试区），符合开发者直觉。
    *   支持深色模式 (Dark Mode)。

## 5. 技术架构选型建议（供参考）

既然你要开发网页版，以下是目前最流行的技术栈建议：

*   **前端：** `React` + `TypeScript` + `TailwindCSS`
    *   *核心组件库：* `Monaco Editor` (微软开源的代码编辑器，用于JSON高亮和编辑，体验极佳)。
*   **后端网关服务：** `Node.js (Express)`
    *   *推荐理由：* Node.js 处理异步I/O（代理转发）性能优秀且全栈统一。
*   **数据库：** 轻量级数据库（如SQLite）用于存储项目、接口配置等数据，预留PostgreSQL和Redis的接入入口，待后续逐步升级后进行中间件替换。

## 6. MVP 版本开发规划 (分步走战略)

如果一下子做全部功能会非常吃力，建议分为三个阶段：

*   **Phase 1 (MVP - 最小可行性产品)：**
    *   实现项目与接口的增删改查。
    *   实现**基础静态Mock功能**（填什么JSON返回什么）。
    *   实现**基础透明转发功能**（将本地 `/api/test` 原封不动转发到 `https://target.com/api`）。
    *   实现基本的 Postman 发送请求测试UI。
*   **Phase 2 (进阶能力核心)：**
    *   引入环境变量体系。
    *   实现请求/响应参数的高级映射与转换（参数注入、字段重命名）。
    *   实现 Faker.js 动态Mock数据。
*   **Phase 3 (团队与商业化准备)：**
    *   完善调用日志分析、QPS统计。
    *   团队协作（多账号、权限分配）。
    *   请求速率限制（Rate Limiting）。

## 7. 验收标准

### 7.1 功能验收
*   能够创建项目并在项目内管理API。
*   能够配置第三方API并进行本地API的转发设置。
*   能够使用静态Mock功能返回预设的JSON数据。
*   能够使用在线调试面板发送请求并查看响应。
*   能够查看请求日志和转发详情。

### 7.2 性能验收
*   转发请求的中间损耗控制在20ms以内。
*   系统能够支持高并发处理。

### 7.3 安全验收
*   系统能够防止SSRF攻击。
*   敏感信息在数据库中加密存储。

### 7.4 易用性验收
*   UI设计符合左右分栏布局，操作直观。
*   支持深色模式切换。
